# 自定义 UDP 视频传输协议 — 技术需求规格说明 v1.1

## 文档信息

- **文档版本**：v1.1
- **适用平台**：Zynq FPGA · PS: ARM Cortex-A53 ×4 · PL: XDMA
- **视频输入**：
  - 4 路可见光 `1600×1200 @ 25fps`
  - 或 4 路红外 `640×480 @ 25fps`
  - 编码格式：H.265（两种模式互斥）
- **网络传输**：千兆以太网 · 自定义 UDP · 每路独立端口（5000~5003）
- **状态**：协议尚未实现，本文档作为开发依据

---

## 1. 背景与范围

本文档定义运行于 FPGA（Zynq 系列）PS 端嵌入式 Linux 系统上的自定义 UDP 视频传输协议的完整技术需求。

协议用于将 4 路 H.265 压缩视频码流从摄像头/编码器传输至 PS，PS 再经 PCIe XDMA 将数据分两路输出：

1. **Raw 透传路**：原始 UDP 包不解析，直接推送至 Host 主机
2. **AI 解析路**：校验帧头，重组完整 AU，解析 NAL，转换 Annex B，VPU 解码后送 PL 侧 AI 推理模块

协议涉及收发两端，双方必须严格遵守本文档中的全部“必须”级需求，方可保证系统正确工作。

> **重要定义**  
> 本文档中的 `frame_seq` 指 **Access Unit（AU）序号**，即“视频帧级序号”，不是单个 NAL Unit 序号。  
> 一个 `frame_seq` 对应一个完整视频帧（AU），一个 AU 内可包含多个 NAL Unit（如 VPS/SPS/PPS/SEI/VCL 等）。

## 2. 系统约束

### 2.1 硬件环境

| 约束项 | 说明 |
|---|---|
| 交换机 | 不支持巨型帧（Jumbo Frame），MTU = 1500 字节 |
| PS DDR | 1 GB，4 路接收 + 两条处理路径共享 |
| PL DDR | 1 GB，AI 推理帧缓存 |
| 网络带宽 | 千兆以太网，4 路 H.265 总带宽 ≤ 30 Mbps，带宽压力极低 |
| 操作系统 | 嵌入式 Linux，裁剪 Ubuntu，标准 socket API |
| PCIe | Xilinx XDMA，DMA 传输至 Host |

### 2.2 H.265 码流约束

H.265 是有状态压缩格式，丢失随机接入帧（IDR / CRA）后，后续预测帧通常无法正常解码，直至下一个有效随机接入点到达方可恢复。因此协议设计必须围绕 **随机接入恢复** 展开。

- I 帧（IDR/CRA）体积通常是 P 帧的 5～10 倍
- 可见光 `1600×1200` 在 `6 Mbps` 码率下，IDR 典型大小约 `30 KB`
- 在标准 MTU 下，一个大 AU 需要拆分为多个 UDP 包发送
- 任何一个关键分片丢失，都可能导致该 AU 不可用

## 3. 协议定义

### 3.1 公共编码规则

1. 所有多字节整数字段在线上传输时，**必须使用网络字节序（big-endian）**
2. 文档中的 `struct` 仅用于字段说明，**禁止**直接依赖本地结构体内存布局收发网络数据
3. 发送端必须逐字段序列化；接收端必须逐字段反序列化
4. 协议版本号升级为 **`0x02`**
5. 任何非 `0x02` 版本的包，接收端必须丢弃并记录日志

### 3.2 协议头结构

每个 UDP 载荷以固定长度协议头开始，头后紧跟 **当前 AU 的分片数据**。

```c
typedef struct __attribute__((packed)) {
    uint16_t magic;             /* 0xAA55，帧头同步魔数                         */
    uint8_t  version;           /* 0x02，协议版本号                             */
    uint8_t  channel_id;        /* 0~3，通道标识                                */
    uint32_t frame_seq;         /* AU序号：每个视频帧/Access Unit递增一次        */
    uint16_t frag_idx;          /* 当前AU分片索引，0-based                      */
    uint16_t frag_total;        /* 当前AU总分片数，>= 1                         */
    uint8_t  is_idr;            /* 1 = 当前AU为随机接入帧（IDR/CRA），0 = 其他    */
    uint8_t  primary_nal_type;  /* 当前AU内首个VCL NAL Unit的类型                */
    uint16_t au_nal_count;      /* 当前AU内NAL Unit总数                          */
    uint64_t pts;               /* 显示时间戳，单位90 kHz                        */
    uint32_t au_size;           /* 当前AU序列化后的总字节数（不含协议头）         */
    uint16_t header_crc;        /* 对前28字节做 CRC-16/IBM                       */
} udp_au_header_t;              /* 固定30字节                                   */
```

> **结论**  
> - 协议头固定长度：**30 字节**
> - `header_crc` 覆盖范围：**前 28 字节**
> - CRC 计算范围：从 `magic` 到 `au_size`

### 3.3 字段说明

| 字段 | 长度 | 说明 |
|---|---:|---|
| `magic` | 2 B | 固定值 `0xAA55`，接收端用于快速识别协议头 |
| `version` | 1 B | 当前协议版本固定为 `0x02` |
| `channel_id` | 1 B | 通道号 `0~3`，与 UDP 端口一一对应 |
| `frame_seq` | 4 B | AU 序号，每个视频帧递增一次，每路独立维护，溢出后自然回绕 |
| `frag_idx` | 2 B | 当前 AU 分片索引，从 `0` 开始 |
| `frag_total` | 2 B | 当前 AU 总分片数，最小值为 `1` |
| `is_idr` | 1 B | 当前 AU 是否为随机接入帧（IDR/CRA） |
| `primary_nal_type` | 1 B | 当前 AU 内首个 VCL NAL 的类型 |
| `au_nal_count` | 2 B | 当前 AU 内的 NAL Unit 总数量，含 VPS/SPS/PPS/SEI/VCL 等 |
| `pts` | 8 B | 显示时间戳，单位 `90 kHz` |
| `au_size` | 4 B | 当前 AU 序列化总长度（不含协议头） |
| `header_crc` | 2 B | 对前 28 字节做 CRC-16/IBM 校验 |

### 3.4 AU 序列化格式

本协议采用 **AU 级封装**。发送端必须先将一个完整 AU 序列化为如下格式，再对其按固定大小分片：

```text
AU_payload := repeat au_nal_count times {
    uint32_t nalu_len_be;   // 4字节，大端
    uint8_t  nalu_bytes[nalu_len_be];
}
```

#### 约束

1. 一个 `frame_seq` 对应一个完整 AU
2. AU 内 NAL Unit 顺序必须与编码器输出顺序一致
3. `au_size` 必须等于整个 `AU_payload` 的总字节数
4. 接收端完成 AU 重组后，按 `nalu_len_be` 逐个解析 NAL
5. AI 解析路在送入 VPU 前，必须为每个 NAL 补 `Annex B` 起始码 `0x00 0x00 0x00 0x01`

### 3.5 UDP 载荷布局与 MTU 规则

每个 UDP 包的载荷由两部分组成：

```text
+--------------------+-----------------------------------+
| udp_au_header_t    | AU 分片数据                        |
| 30 bytes           | <= 1440 bytes                     |
+--------------------+-----------------------------------+
```

#### 长度计算

- 以太网 MTU = `1500`
- IPv4 头 = `20 B`
- UDP 头 = `8 B`
- 因此 UDP payload 最大 = `1472 B`
- 扣除协议头 `30 B`
- **理论最大 AU 分片数据长度 = `1442 B`**
- 为统一实现并保留余量，协议规定：
  - **标准分片长度固定为 `1440 B`**
  - 除最后一个分片外，其他分片数据长度必须为 `1440 B`

#### 分片计算公式

```text
frag_total = ceil(au_size / 1440)
frag_idx   = 0 ... frag_total-1
```

### 3.6 CRC 参数定义

`header_crc` 必须使用 **CRC-16/IBM**，参数固定如下：

- poly = `0x8005`
- init = `0x0000`
- refin = `true`
- refout = `true`
- xorout = `0x0000`

接收端与发送端必须严格使用同一参数集合，禁止仅按“CRC-16/IBM”名称自行选库默认值。

### 3.7 `frame_seq` 回绕比较规则

`frame_seq` 为 `uint32_t`，允许自然回绕。接收端必须按模 `2^32` 比较新旧，推荐使用如下规则：

```c
static inline bool seq_newer(uint32_t a, uint32_t b) {
    return (int32_t)(a - b) > 0;
}
```

不得将 `0xFFFFFFFF -> 0x00000000` 误判为大幅回退。

## 4. 发送端技术需求

### 4.1 AU 封装与分片规则

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| TX-01 | 必须以 **Access Unit（视频帧）** 为最小传输单位；一个 `frame_seq` 只对应一个 AU | 必须 |
| TX-02 | 必须先将整个 AU 序列化为 `[4B nalu_len_be][nalu_bytes]...` 格式，再进行应用层分片 | 必须 |
| TX-03 | 分片边界允许落在 AU 内任意字节偏移处；NAL Unit 边界由 `nalu_len_be` 描述，不要求分片边界与 NAL 边界重合 | 必须 |
| TX-04 | `au_size` 必须填写当前 AU 序列化后的总字节数（不含协议头） | 必须 |
| TX-05 | `frag_total` 必须按 `ceil(au_size / 1440)` 计算 | 必须 |
| TX-06 | 同一 AU 的分片必须按 `frag_idx` 顺序连续发送，不得插入其他 AU 的分片 | 必须 |
| TX-07 | 同一 AU 的所有分片除 `frag_idx` 与 `header_crc` 外，头字段必须保持一致 | 必须 |
| TX-08 | 除最后一个分片外，每个分片的数据长度必须为 `1440 B` | 必须 |

### 4.2 IDR / CRA 与参数集规则

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| TX-09 | 当 AU 为随机接入帧（`IDR_W_RADL=19`、`IDR_N_LP=20`、`CRA_NUT=21`）时，`is_idr` 必须置 `1`；否则置 `0` | 必须 |
| TX-10 | 当 `is_idr = 1` 时，该 AU 内必须包含完整的 `VPS + SPS + PPS + IRAP 图像 NAL` | 必须 |
| TX-11 | IDR/CRA AU 内 NAL 顺序必须为 `VPS -> SPS -> PPS -> (可选SEI) -> 首个VCL(IRAP)` | 必须 |
| TX-12 | `primary_nal_type` 必须填写当前 AU 内首个 VCL NAL 的类型 | 必须 |
| TX-13 | `au_nal_count` 必须准确填写当前 AU 内 NAL Unit 的实际数量 | 必须 |
| TX-14 | GOP 长度建议配置为不超过 `25` 帧（即 1 秒一个随机接入点 @ 25fps） | 应该 |
| TX-15 | 大 AU（尤其是 IDR/CRA）建议在一个帧周期内进行平滑发送（Pacing），避免短时突发冲击交换机缓存 | 应该 |
| TX-16 | 当检测到编码参数变更（分辨率、帧率、GOP、码率模式等）时，应立即触发一次完整的 `VPS/SPS/PPS + IDR/CRA` AU | 应该 |

### 4.3 帧头填写规则

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| TX-17 | `magic` 必须固定填写 `0xAA55` | 必须 |
| TX-18 | `version` 必须固定填写 `0x02` | 必须 |
| TX-19 | `channel_id` 必须与发送端口对应：5000→0，5001→1，5002→2，5003→3 | 必须 |
| TX-20 | `frame_seq` 必须按 AU 为粒度单调递增，每路独立计数，溢出后自然回绕 | 必须 |
| TX-21 | `pts` 应从编码器获取真实显示时间戳，以 `90 kHz` 单位填写 | 应该 |
| TX-22 | `header_crc` 必须对头部前 `28` 字节按本协议定义计算 | 必须 |

### 4.4 网络发送规则

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| TX-23 | 4 路视频必须通过 4 个独立 UDP 端口发送：通道 0→5000，1→5001，2→5002，3→5003 | 必须 |
| TX-24 | 发送端不得使用 IP 分片，所有分片必须在应用层完成 | 必须 |
| TX-25 | 发送端必须保证 `UDP payload <= 1472 B` | 必须 |
| TX-26 | 发送端必须保证单包 AU 分片数据长度 `<= 1440 B` | 必须 |
| TX-27 | 发送端应开启 DF（Don't Fragment）语义，任何超 MTU 报文都应视为实现错误 | 应该 |
| TX-28 | 禁止直接发送本地结构体二进制镜像；必须按字段序列化为网络字节序 | 必须 |

## 5. 接收端技术需求

### 5.1 socket 与线程架构

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-01 | 必须为每个端口创建独立 UDP socket，分别绑定端口 5000、5001、5002、5003 | 必须 |
| RX-02 | 必须为每个 socket 创建独立接收线程，4 路完全隔离 | 必须 |
| RX-03 | 接收线程应绑定固定 CPU 核心（建议 CPU0～CPU3 各绑一路） | 应该 |
| RX-04 | 每个 socket 的 `SO_RCVBUF` 必须不小于 8 MB，建议设置为 32 MB | 必须 |
| RX-05 | 系统参数 `net.core.rmem_max` 必须与 socket 缓冲设置同步调整 | 必须 |
| RX-06 | 网卡接收 Ring Buffer 应通过 `ethtool` 扩展至 4096 | 应该 |

### 5.2 协议头与边界校验

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-07 | 收到 UDP 包后必须先校验 `magic == 0xAA55`，否则立即丢弃 | 必须 |
| RX-08 | 必须校验 `version == 0x02`，未知版本立即丢弃并记录日志 | 必须 |
| RX-09 | 必须校验 `channel_id` 与 socket 绑定端口对应关系一致，不一致立即丢弃 | 必须 |
| RX-10 | 必须校验 `header_crc`，失败立即丢弃 | 必须 |
| RX-11 | 必须校验 `frag_total >= 1` | 必须 |
| RX-12 | 必须校验 `frag_idx < frag_total` | 必须 |
| RX-13 | 必须校验 `au_nal_count >= 1` | 必须 |
| RX-14 | 必须校验 `au_size >= 4 * au_nal_count` | 必须 |
| RX-15 | 必须校验 `au_size <= 200 KB` | 必须 |
| RX-16 | 任一上述条件失败时，必须丢弃该包并记录告警 | 必须 |

### 5.3 AU 重组

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-17 | 必须以 `(channel_id, frame_seq)` 为键维护每路独立的 AU 重组缓冲 | 必须 |
| RX-18 | AU 重组 buffer 深度建议不少于 4 帧 | 应该 |
| RX-19 | 当前包数据写入偏移必须按 `offset = frag_idx * 1440` 计算 | 必须 |
| RX-20 | 当前包数据段长度 `payload_len = udp_payload_len - 30` | 必须 |
| RX-21 | 非最后一个分片必须满足 `payload_len == 1440` | 必须 |
| RX-22 | 最后一个分片必须满足 `1 <= payload_len <= 1440` | 必须 |
| RX-23 | 必须校验 `offset + payload_len <= au_size` | 必须 |
| RX-24 | 同一 AU 内允许分片乱序到达，接收端必须按 `frag_idx` 放置到对应偏移 | 必须 |
| RX-25 | 同一 `frag_idx` 的重复包必须丢弃并计数，不得覆盖已写入数据 | 必须 |
| RX-26 | 收齐 `frag_total` 个唯一分片后，方可认定当前 AU 完整 | 必须 |
| RX-27 | 从收到某 AU 第一个分片起，超过 `80 ms` 仍未收齐，必须清空该 AU 缓冲并标记超时丢弃 | 必须 |
| RX-28 | 当检测到 `frame_seq` 跳变导致 AU 缺失时，必须立即标记缺失帧，不等待缺失 AU | 必须 |
| RX-29 | `frame_seq` 跳变判断必须使用模 `2^32` 比较规则，不得直接做无符号大小比较 | 必须 |

### 5.4 完整 AU 解析

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-30 | 完成 AU 重组后，必须校验重组总字节数恰好等于 `au_size` | 必须 |
| RX-31 | 必须按 `au_nal_count` 逐个解析 `[4B nalu_len_be][nalu_bytes]` 结构 | 必须 |
| RX-32 | 若任意 NAL 长度越界、解析提前结束或解析后偏移不等于 `au_size`，必须丢弃该 AU | 必须 |
| RX-33 | 若 AU 完整但解析失败，必须记录错误并将该通道置为“等待 IDR”状态 | 必须 |

### 5.5 IDR 恢复机制

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-34 | AI 解析路在以下任一事件后，必须将对应通道置为“等待 IDR”状态：AU 缺失、重组超时、AU 解析失败、VPU 解码报错 | 必须 |
| RX-35 | 处于“等待 IDR”状态时，必须丢弃该通道后续所有非 IDR 完整 AU | 必须 |
| RX-36 | 收到 `is_idr = 1` 的完整 AU 后，必须验证该 AU 内至少包含 `VPS`、`SPS`、`PPS` 与一个 IRAP 图像 NAL | 必须 |
| RX-37 | 若 IDR AU 缺少参数集或首个随机接入 VCL，必须继续保持“等待 IDR”状态 | 必须 |
| RX-38 | 恢复解码前，必须先重置该通道 VPU 解码上下文 | 必须 |
| RX-39 | 恢复解码时，必须按 AU 内原始顺序将全部 NAL（转换为 Annex B 后）送入 VPU | 必须 |
| RX-40 | IDR 恢复必须按通道独立进行，4 路互不影响 | 必须 |
| RX-41 | 接收端应缓存最近一次成功解析的 VPS/SPS/PPS，用于辅助诊断与恢复策略增强 | 应该 |

### 5.6 Annex B 格式转换

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-42 | AI 解析路在将 NAL 送入 VPU 前，必须为每个 NAL 前补 `0x00 0x00 0x00 0x01` 起始码 | 必须 |
| RX-43 | Annex B 转换必须严格按 AU 内 NAL 顺序执行 | 必须 |
| RX-44 | Raw 透传路不得做 Annex B 转换 | 必须 |

### 5.7 双路数据路径解耦

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-45 | Raw 透传路和 AI 解析路必须使用独立处理线程，不得共用同一线程 | 必须 |
| RX-46 | Raw 透传路在完成协议头校验后，应立即将原始 UDP 包（含协议头）提交 XDMA 队列 | 必须 |
| RX-47 | AI 解析路在 AU 重组与 Annex B 转换完成后，将完整 AU 提交 VPU 解码 | 必须 |
| RX-48 | 两路之间应通过共享内存 Ring Buffer 传递数据，避免额外内存拷贝 | 应该 |
| RX-49 | Raw 路与 AI 路任一路阻塞，不得反向阻塞接收线程的基本收包能力 | 必须 |

### 5.8 背压与队列满处理

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-50 | 当 Raw 透传队列满时，必须采用明确丢包策略，禁止阻塞接收线程无限等待 | 必须 |
| RX-51 | 当 AI 解码队列满时，必须采用明确丢帧策略，禁止阻塞接收线程无限等待 | 必须 |
| RX-52 | 建议优先保留较新的完整 IDR/CRA AU，优先丢弃过期非 IDR AU | 应该 |
| RX-53 | 队列满、丢包、丢帧事件必须进入统计计数器 | 必须 |

### 5.9 可见光 / 红外模式处理

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| RX-54 | 可见光模式（`1600×1200`）下，VPU 解码后的 YUV 帧必须 resize 到 `800×576` 后写入 PL DDR | 必须 |
| RX-55 | 红外模式（`640×480`）下，VPU 解码后的 YUV 帧直接写入 PL DDR，不做 resize | 必须 |
| RX-56 | 工作模式切换（可见光 ↔ 红外）时，必须重置全部 4 路的 VPU 解码上下文与 AU 重组缓冲 | 必须 |
| RX-57 | 模式切换时，必须切换为新模式参数并丢弃旧模式残留 AU | 必须 |

## 7. 性能需求

| 指标 | 要求 | 说明 |
|---|---|---|
| 端到端延迟 | `≤ 100 ms` | 接收 → 写入 PL DDR，单路，不含网络传输延迟 |
| 丢帧恢复 | `≤ 2 秒` | GOP = 25 帧时，从异常到恢复正常解码 |
| CPU 占用 | 4 路总接收 + 解析 `≤ 70%` | 4 核 A53 合计 |
| PS DDR 占用 | 接收 buffer + 重组 buffer `≤ 256 MB` | 4 路合计峰值 |
| XDMA 吞吐 | Raw 透传路吞吐 `≥ 60 Mbps` | 4 路 H.265 满载 |

## 8. 错误处理与日志需求

| 编号 | 需求描述 | 优先级 |
|---|---|---|
| ER-01 | 接收端必须维护每路统计计数器：收包总数、CRC 失败数、版本错误数、非法包数、重复分片数、AU 重组超时数、`frame_seq` 跳变次数、等待 IDR 次数、VPU 解码失败次数、队列满丢弃次数 | 必须 |
| ER-02 | 以上统计计数器必须通过调试接口（如 `/proc` 或 `sysfs`）可读 | 必须 |
| ER-03 | CRC 校验失败、版本未知、`channel_id` 不匹配、非法包结构等异常，必须记录 `LOG_WARNING`，并包含端口号、`frame_seq`、错误类型 | 必须 |
| ER-04 | 同类错误日志必须限速：每秒最多记录 10 条，防止日志风暴 | 必须 |
| ER-05 | 进入“等待 IDR”状态时，必须记录 `LOG_WARNING`，注明通道号和触发原因（缺帧 / 超时 / 解析失败 / VPU错误） | 必须 |
| ER-06 | 恢复正常解码后，必须记录 `LOG_INFO`，注明通道号与恢复耗时（ms） | 必须 |
| ER-07 | 重组 buffer 或临时内存分配失败时，必须记录 `LOG_ERROR` 并跳过该 AU，不得 crash | 必须 |
| ER-08 | 队列满导致的丢包/丢帧必须进入计数器并可被监控 | 必须 |

## 9. 测试与验收标准

### 9.1 单路基础连通测试

1. 发送端发送单路 H.265 码流，接收端成功完成：
   - UDP 收包
   - 协议头校验
   - AU 重组
   - AU 解析
   - Annex B 补齐
   - VPU 解码
2. 解码输出 YUV 帧无花屏、无明显帧序异常
3. Raw 透传路将原始 UDP 包完整写入 XDMA，Host 端可识别协议头并提取数据

### 9.2 AU 解析正确性测试

1. 构造包含 `VPS/SPS/PPS/SEI/VCL` 的 AU，验证 `au_nal_count` 与 `au_size` 解析一致
2. 构造跨 NAL 边界分片的 AU，验证接收端仍可正确重组并解析
3. 验证 `primary_nal_type` 与首个 VCL NAL 一致

### 9.3 丢帧恢复测试

1. 人为丢弃某一路第 N 个 AU 的一个或多个分片
2. 验证接收端在 `80 ms` 超时或跳变检测后进入“等待 IDR”状态
3. 验证下一个完整 IDR/CRA AU 到达后，系统可恢复解码
4. 恢复时间必须满足 `≤ 2 秒`
5. 其他 3 路不受影响

### 9.4 乱序与重复包测试

1. 在同一 AU 内打乱分片到达顺序，验证可正确重组
2. 重复发送同一 `frag_idx`，验证重复包被丢弃且计数正确
3. 若跨 AU 到达顺序异常，验证系统按既定策略处理并记录日志

### 9.5 协议健壮性测试

1. 错误 `channel_id`：应丢弃并记录告警
2. 错误 `header_crc`：应丢弃，不进入重组
3. 错误 `version = 0xFF`：应丢弃并记录日志
4. `frag_total = 0`：应丢弃
5. `frag_idx >= frag_total`：应丢弃
6. `au_nal_count = 0`：应丢弃
7. `au_size` 超出上限：应丢弃
8. AU 可完整重组但内部 `nalu_len` 越界：应丢弃并进入“等待 IDR”

### 9.6 参数集与随机接入测试

1. 构造 `is_idr = 1` 但缺少 VPS/SPS/PPS 的 AU，验证不得退出“等待 IDR”
2. 构造完整 `VPS/SPS/PPS + IDR` AU，验证可恢复
3. 模拟编码参数变化，验证发送端主动推送新的参数集 AU，接收端可重新建立解码上下文

### 9.7 `frame_seq` 回绕测试

1. 发送接近 `0xFFFFFFFF` 的 AU 序列
2. 验证 `0xFFFFFFFF -> 0x00000000` 后接收端仍可正确判定新旧关系
3. 不得因回绕误判为大量丢帧

### 9.8 4 路并发压力测试

1. 4 路视频同时接收，持续运行 30 分钟
2. 统计丢帧率、CPU 占用、PS DDR 占用，必须满足第 7 章指标
3. 可见光与红外两种模式各测试一次

### 9.9 长稳测试

1. 4 路视频连续运行 `24 小时`
2. 验证无内存泄漏、无计数器异常增长、无线程卡死、无不可恢复状态
3. 出现偶发异常后必须可在下一个有效随机接入点恢复

## 10. 术语表

| 术语 | 定义 |
|---|---|
| Access Unit（AU） | 一个完整视频帧对应的编码访问单元，可包含多个 NAL Unit |
| IDR / CRA | H.265 随机接入帧，解码器可在此处恢复 |
| NAL Unit | Network Abstraction Layer Unit，H.265 基本封装单元 |
| VPS / SPS / PPS | H.265 参数集，解码随机接入帧的前提 |
| Annex B | H.265 字节流格式，每个 NAL 前添加 `0x00000001` 起始码 |
| GOP | Group of Pictures，一个随机接入帧及其后续预测帧组成的编码单元 |
| PTS | Presentation Timestamp，显示时间戳，单位 90 kHz |
| XDMA | Xilinx DMA，PL 侧 PCIe DMA 控制器 |
| CRC-16/IBM | 头部完整性校验算法，本协议固定参数见第 3.6 节 |

## 附录 B：需求优先级说明

本文档使用三级优先级：

1. **必须**：强制要求，收发两端都必须实现，缺失则系统功能不正确
2. **应该**：强烈建议，实现后可显著提升系统稳定性或性能
3. **可以**：可选优化项，在资源允许时实现

> **实施建议**  
> 建议优先实现全部“必须”级需求并通过第 9 章验收测试后，再推进“应该”级优化，最后处理“可以”级扩展。
