# 自定义 UDP 视频传输协议 — 发送端技术需求规格说明（测试版）v0.1

## 文档信息

- **文档名称**：自定义 UDP 视频传输协议 — 发送端技术需求规格说明（测试版）
- **文档版本**：v0.1
- **适用阶段**：联调 / 协议验证 / 接收端联测
- **实现形态**：FFmpeg 编码 + Python/C 封装 + UDP 发送
- **对应协议版本**：`v1.1 / version = 0x02`
- **目标定位**：先实现“能稳定产出符合协议定义的测试码流”，不以最终量产实现方式为约束

---

## 1. 背景与目标

为尽快开展接收端联调与协议验证，发送端第一阶段不直接采用最终嵌入式发送实现，而采用以下测试架构：

1. 使用 **FFmpeg** 完成视频采集/读取与 H.265 编码
2. 使用 **Python 或 C** 进程读取 FFmpeg 输出的 H.265 码流
3. 在封装进程中完成：
   - Annex B 码流解析
   - AU（Access Unit）边界识别
   - AU 序列化
   - 自定义协议头填充
   - 应用层分片
   - UDP 发包
4. 用于驱动接收端完成：
   - 协议头校验
   - AU 重组
   - Annex B 恢复
   - VPU 解码
   - Raw 透传验证

本阶段目标是 **验证协议正确性与系统联通性**，而不是追求最终极致性能。

---

## 2. 范围

本文件仅定义**发送端测试实现**的技术需求，不覆盖以下内容：

- 最终板端摄像头驱动接入细节
- 最终 FPGA/SoC 内部硬件发送链路

- UDP 分片与发送规则
- 发送端日志、统计、错误处理
- 联调验收标准

## 3. 总体架构

### 3.1 逻辑架构

```text
视频源 / 文件 / RTSP / 摄像头
            |
            v
        FFmpeg 编码器
            |
            | H.265 Annex B Elementary Stream
            v
   Sender Packager（Python/C）
     - 解析 NAL Unit
     - 识别 Access Unit
     - 识别 IDR/CRA
     - 生成 AU_payload
     - 填写协议头
     - 分片
     - UDP 发送
            |
            v
     目标接收端（4路独立端口）
```

### 3.2 进程划分

测试阶段建议采用 **双进程模型**：

- **进程 A：FFmpeg**
  - 负责输入读取与 H.265 编码
  - 输出原始 H.265 码流到 `stdout` / pipe / named pipe / 本地 socket
- **进程 B：封装发送器（Python/C）**
  - 负责读取码流并完成协议发送

> 建议优先使用“FFmpeg stdout + Python 封装器”快速验证协议；
> 在协议跑通后，将封装器核心迁移到 C，以提升稳定性和定时精度。

---

## 4. 测试阶段设计原则

1. **协议正确性优先于性能最优**
2. **行为可观测优先于实现优雅**
3. **先单路跑通，再扩展到 4 路**
4. **发送端必须稳定产出“可被接收端正确恢复”的 IDR AU**
5. **不得因测试实现简化而偏离协议 v1.1 关键定义**

## 5. 输入与编码要求

### 5.1 输入源要求

发送端测试实现应支持以下输入源中的一种或多种：

- 本地视频文件
- USB / V4L2 摄像头
- RTSP 码流
- 预先准备的 YUV / MP4 / TS 文件

测试阶段优先建议：

1. **文件输入**：便于结果复现
2. **本地摄像头输入**：便于实时链路验证
3. **RTSP 输入**：便于模拟网络前级输入

### 5.2 编码器要求

发送端测试实现必须输出 **H.265 Elementary Stream**，并满足以下要求：

- 编码输出应为连续 H.265 码流
- 应能稳定产生 IDR/CRA 随机接入帧
- GOP 建议固定为 **25 帧**
- 建议关闭随机场景切换导致的不可控关键帧间隔抖动
- 应保证关键帧前可获得完整参数集（VPS/SPS/PPS）
- 输出应便于封装器识别 AU 边界

### 5.3 编码参数建议

测试阶段建议采用以下参数策略：

- 编码器：`libx265`（优先）
- 码率：先用固定目标码率，避免调试期码流剧烈波动
- GOP：`25`
- B 帧：建议关闭，优先降低时序复杂度
- 随机接入点：固定周期产生
- 参数集：关键帧前重复输出
- AUD：建议开启，便于 AU 边界识别（若环境支持）
- 低延迟：建议启用零延迟/低缓存策略

> 测试阶段的核心目标是 **让封装器稳定识别 Access Unit**，而不是追求最佳画质压缩比。

## 6. 发送端模块划分

### 6.1 模块列表

发送端封装器至少应包含以下模块：

1. **码流输入模块**
   - 从 FFmpeg 管道读取字节流
2. **NAL 解析模块**
   - 识别 Annex B 起始码
   - 切分出完整 NAL Unit
3. **AU 组帧模块**
   - 根据 H.265 语义判断 AU 边界
   - 将多个 NAL 归并为一个 AU
4. **协议封装模块**
   - 构造 `AU_payload`
   - 填写协议头
   - 生成分片
5. **UDP 发送模块**
   - 按分片顺序发送
   - 控制节奏与统计
6. **日志与统计模块**
   - 输出发送端运行状态
   - 提供调试计数器
7. **配置模块**
   - 支持通过命令行/配置文件选择通道号、目标 IP、端口、码率、GOP 等

### 6.2 Python 与 C 的职责建议

#### Python 版本（建议先实现）

适合完成：

- 协议正确性验证
- AU 解析验证
- 单路/双路联调
- 抓日志与快速改逻辑

#### C 版本（建议第二阶段实现）

适合完成：

- 更稳定的定时发送
- 更低抖动
- 更低 CPU 占用
- 多路长期运行测试

## 7. Annex B 解析要求

### 7.1 输入格式

封装器读取到的 H.265 码流应视为 **Annex B 字节流**，即 NAL Unit 之间通过起始码分隔：

- `0x00 00 01`
- 或 `0x00 00 00 01`

### 7.2 NAL Unit 提取

封装器必须能够：

1. 正确识别 3 字节与 4 字节起始码
2. 切分出完整 NAL Unit
3. 去除起始码后，将纯 NAL 字节内容传给 AU 组帧模块
4. 获取 NAL Unit Type，用于识别：
   - VPS
   - SPS
   - PPS
   - SEI
   - AUD
   - VCL（含 IDR / CRA / 普通图像）

### 7.3 错误输入处理

当出现以下情况时，封装器必须记录错误并按策略处理：

- 起始码异常
- 连续空 NAL
- NAL 长度为 0
- 无法识别的 NAL Type
- 长时间未形成完整 AU

测试阶段允许“丢弃异常 NAL 并继续运行”，但不得 silent fail。

---

## 8. Access Unit（AU）识别要求

### 8.1 AU 传输语义

发送端必须严格采用 **方案 B：以 Access Unit 为单位传输**。

即：

- 一个 `frame_seq` 对应一个视频帧
- 一个 AU 内可包含多个 NAL Unit
- AU 中可包含：
  - VPS
  - SPS
  - PPS
  - SEI
  - AUD
  - 一个或多个 VCL NAL

### 8.2 AU 组帧规则

发送端必须在封装层将属于同一视频帧的 NAL Unit 归并为一个 AU，然后再进行协议封装。

建议组帧顺序为：

```text
[AUD] [VPS] [SPS] [PPS] [SEI] [VCL...]
```

> 若输入中不存在 AUD，封装器必须仍可工作，但应通过 VCL 边界规则完成 AU 划分。

### 8.3 IDR / CRA 识别

当 AU 内首个随机接入图像 NAL 类型属于：

- `IDR_W_RADL = 19`
- `IDR_N_LP = 20`
- `CRA_NUT = 21`

则该 AU 的 `is_idr` 必须置为 `1`。

否则置为 `0`。

### 8.4 参数集注入要求

测试发送端必须满足：

- 每个 `is_idr = 1` 的 AU 中必须包含完整 `VPS + SPS + PPS`
- 不得将参数集放在独立 `frame_seq` 中单独发送
- 若编码器未在关键帧前重复输出参数集，封装器应具备“缓存最近一次 VPS/SPS/PPS 并在 IDR AU 前补齐”的能力

> 测试阶段建议：**封装器实现参数集缓存与注入能力**。

---

## 9. AU 序列化要求

在形成完整 AU 后，发送端必须按协议 v1.1 将 AU 序列化为：

```text
AU_payload := repeat au_nal_count times {
    uint32_t nalu_len_be;
    uint8_t  nalu_bytes[nalu_len_be];
}
```

### 要求

1. `nalu_len_be` 必须为 **4 字节大端**
2. `nalu_bytes` 不含 Annex B 起始码
3. `au_nal_count` 必须准确填写
4. `au_size` 必须等于全部 `4B长度前缀 + NAL内容` 的总长度

---

## 10. 协议头填写要求

发送端必须填写如下头字段：

| 字段 | 要求 |
|---|---|
| `magic` | 固定 `0xAA55` |
| `version` | 固定 `0x02` |
| `channel_id` | 与发送端口一致 |
| `frame_seq` | 以 AU 为粒度递增 |
| `frag_idx` | 当前 AU 分片索引 |
| `frag_total` | 当前 AU 总分片数 |
| `is_idr` | 当前 AU 是否为随机接入帧 |
| `primary_nal_type` | 当前 AU 内首个 VCL NAL 的类型 |
| `au_nal_count` | 当前 AU 中 NAL 数量 |
| `pts` | 当前 AU 显示时间戳，90kHz |
| `au_size` | 当前 AU 序列化总长度 |
| `header_crc` | 对前 28 字节做 CRC-16/IBM |

### 关键约束

1. 所有多字节字段必须使用 **big-endian**
2. `frame_seq` 必须每个 AU 自增一次
3. 同一 AU 的全部分片，除 `frag_idx` 和 `header_crc` 外，其他字段必须完全一致
4. 禁止直接发送本地结构体内存镜像，必须手工序列化

---

## 11. 分片与发送要求

### 11.1 分片规则

发送端必须按以下规则分片：

- 协议头固定 `30 B`
- 单包 AU 数据净载荷上限固定 `1440 B`
- 分片总数计算方式：

```text
frag_total = ceil(au_size / 1440)
```

### 11.2 分片顺序

同一 AU 的分片必须：

- 按 `frag_idx = 0 ... frag_total-1` 顺序发送
- 不得插入其他 AU 的分片
- 不得乱序发送

### 11.3 发送节奏

测试发送端应支持两种节奏模式：

#### 模式 A：按编码实时节奏发送

适用于实时联调：

- 按 FFmpeg 实时输出节奏直接发送
- 或根据 PTS 做轻量 pacing

#### 模式 B：离线文件回放节奏发送

适用于可重复测试：

- 支持按源帧率发送
- 支持倍速 / 降速 / 单步调试

### 11.4 突发控制

对于大 IDR AU，建议支持基础 pacing，避免在极短时间内集中打出全部分片。

测试阶段可采用简单策略：

- 在同一帧周期内均匀发送
- 或在分片之间插入微小 sleep（例如 0.2~1ms 量级，可配置）

---

## 12. PTS 与时序要求

### 12.1 PTS 来源

发送端应优先使用 FFmpeg 输出帧的原始时间信息构造 `pts`。

若无法直接获得，可按固定帧率推导：

- `25fps` 时，帧间隔 = `3600`（90kHz 时钟）

### 12.2 PTS 填写规则

- `pts` 必须为每路独立时间轴
- 必须单调递增
- 不得长期填 0 或固定值
- 允许测试阶段采用“基于帧序号换算”的方式填充，但实现中必须明确标注

## 14. 发送端日志与统计要求

发送端必须至少输出以下统计信息：

- 已发送 AU 数
- 已发送 UDP 包数
- 已发送字节数
- IDR AU 数
- 参数集注入次数
- CRC 计算次数

- AU 解析失败次数
- 空 AU / 非法 AU 次数
- 平均每 AU 分片数
- 当前 `frame_seq`
- 当前 `pts`

### 日志级别建议

- `INFO`：启动、停止、配置、IDR 发送
- `WARNING`：参数集缺失、异常 NAL、PTS 回退、AU 解析异常
- `ERROR`：pipe 读取失败、socket 发送失败、CRC 处理异常、内部状态机错误

## 15. 错误处理策略

### 15.1 FFmpeg 异常

当 FFmpeg 子进程退出、编码中断或 pipe 断开时：

- 封装器必须记录错误
- 可选择退出，或按配置自动重启 FFmpeg
- 不得静默卡死

### 15.2 AU 异常

当出现以下情况时，当前 AU 必须丢弃并计数：

- 无 VCL NAL
- `au_nal_count = 0`
- `au_size = 0`
- IDR AU 缺少参数集且无法补齐
- NAL 长度序列化失败

### 15.3 网络异常

UDP 发送失败时：

- 必须记录错误码和失败分片位置
- 测试阶段允许继续后续 AU 发送
- 不要求重传

## 20. 测试与验收标准

### 20.1 单路基础验证

1. 发送端可从文件或摄像头读取输入
2. FFmpeg 可持续输出 H.265 码流
3. 封装器可正确切分 NAL
4. 封装器可正确组装 AU
5. 接收端可成功重组并解码
6. IDR 恢复机制可工作

### 20.2 协议正确性验证

1. 抓包检查协议头字段正确
2. 检查：
   - `magic`
   - `version`
   - `channel_id`
   - `frame_seq`
   - `frag_idx`
   - `frag_total`
   - `is_idr`
   - `primary_nal_type`
   - `au_nal_count`
   - `pts`
   - `au_size`
   - `header_crc`
3. 验证大 AU 分片顺序正确
4. 验证 IDR AU 含 VPS/SPS/PPS

### 20.3 接收端联调验证

1. 接收端收到完整 AU 后可恢复 Annex B
2. 可成功送入 VPU 解码
3. 丢弃某个关键 AU 后，可在下一个 IDR AU 恢复
4. Raw 路可收到完整原始 UDP 包

### 20.4 长稳测试

1. 单路连续发送 30 分钟无异常退出
2. 多路发送过程中无明显 `frame_seq` 跳号异常
3. 无持续性的参数集丢失
4. 无大量发送失败

## 23. 实施建议
